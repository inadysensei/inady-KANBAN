# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Language

This repository's language is **English**. Write all code, comments, and issues in English.

## What this is

A localhost, single-user inady KANBAN board that drives the **Cursor / Claude / Cline CLI** (`cursor`, `claude`, or `cline`) from the browser. Tickets flow To Do → Doing → WIP → Done. Starting an agent on a ticket auto-moves it to Doing; WIP is a parking lot for items set aside, and resuming work on a WIP ticket (typing into a resumed session, re-running a session, or launching a new one) auto-moves it back to Doing — merely opening a session to read it is glance-safe and keeps the ticket parked; the user drags to Done manually. Each ticket holds multiple agent sessions, each running `cursor`, `claude`, or `cline` in a server-side PTY streamed to an in-browser xterm.js terminal over a WebSocket.

## Architecture — the load-bearing ideas

### Two module graphs, one process
`server.ts` (run by `tsx`) and Next's bundled Server Actions/components are **separate module graphs in the same Node process**. This drives several rules:

- **DB connection is a shared singleton.** `src/db/client.ts` exports a lazy `Proxy` over a `globalThis`-cached better-sqlite3 handle, so both graphs share one connection. The proxy is *lazy on purpose*: opening at import time made `next build`'s parallel page-data workers contend (`SQLITE_BUSY`). Don't make `db` eager.
- **The PTY registry Map is NOT shared.** Only `server.ts` (via `src/lib/pty-registry.ts`) ever touches live PTYs. **Server Actions must never import `pty-registry`** — they'd get a different, empty Map (and pull `node-pty` into the wrong bundle). PTY lifecycle is driven entirely over the WebSocket.
- **The board event bus IS shared** (`src/lib/board-events.ts`): a `globalThis`-cached `EventEmitter` (same pattern as the db handle) carrying a `BoardEvent` discriminated union (`kind`). `pty-registry` publishes `SessionEvent`s (running/finished/error/killed plus hook-driven *activity* events carrying `activity`); `ticket-core`'s `insertTicket`/`updateTicketFields` publish a `TicketEvent` on every create/update — so the board UI Server Action, the HTTP API, *and* the MCP broadcast through the one shared write path. This is what refreshes **other** open tabs: `revalidatePath` only updates the tab that ran the action (and external `POST`/`PATCH` writes have no request context to revalidate at all), so the bus is the only thing that keeps a second tab in sync. (Delete is deliberately *not* on the bus: it's only reachable in-app — the icebox page / Server Actions, which `revalidatePath` the acting tab — so a second tab won't drop a deleted row until its next refresh; `TicketEvent.action` is `"created" | "updated"` only and would need widening if delete is ever broadcast.) The SSE endpoint in `server.ts` fans every event out to every tab, where `NotificationCenter` (layout-mounted) refreshes the route on **all** frames, but only `SessionEvent`s reach the notify/badge path — it fires desktop notifications (including an "agent needs your input" one on `activity: "awaiting"`) and keeps an **unread badge** (count of notify-worthy events received while the tab is hidden, reset on focus) reflected on the favicon (canvas), the tab title prefix, the OS app badge (`navigator.setAppBadge`, best-effort), and the bell — independent of notification permission; `TicketEvent`s refresh only. The pure count math lives in `src/lib/notification-display.ts` (`shouldNotify`/`notificationBody`/`nextUnreadCount`/`formatBadgeCount`, unit-tested, session-only — the component narrows on `kind` first); the DOM side effects stay in the component. Clients import only the `BoardEvent` *type* — `node:events` must never enter the client bundle.
- **Import convention is deliberate**: `src/db/**` and `src/lib/**` use **relative imports** (loaded by both `tsx` and Next); `src/actions`, `src/app`, `src/components` use the `@/` alias (Next-bundled only). Keep `node-pty` / `better-sqlite3` out of the client bundle — they're declared in `serverExternalPackages` (next.config.mjs).

### WebSocket terminal (`server.ts` + `src/lib/terminal-protocol.ts`)
- One endpoint: `/ws/terminal/:sessionDbId`. Protocol is JSON frames — client `start`/`stdin`/`resize`/`kill` ↔ server `ready`/`replay`/`stdout`/`exit`/`error`.
- HTTP control plane (custom server only, not Server Actions): `POST /api/agent-sessions/:id/kill`, `POST /api/agent-sessions/:id/activity/(awaiting|busy)` (hook-driven activity overlay — see below), `POST /api/tickets/:id/kill-sessions`, `POST /api/tickets` (create a ticket from outside the Next app — e.g. an import script — via the shared `insertTicket`; body capped, does NOT `revalidatePath` — `insertTicket`'s `TicketEvent` is what auto-refreshes open boards), `GET /api/tickets` (list, optional `?status=`, via `listTickets`), `GET /api/tickets/:id` (read one, via `getTicket`), `PATCH /api/tickets/:id` (update fields, via `updateTicketFields`, whose `TicketEvent` likewise auto-refreshes open boards; no `revalidatePath`), `GET /api/agent-sessions/live-count`, `GET /api/events` (SSE stream of `BoardEvent`s; writes are guarded against destroyed responses), and `POST /mcp` (MCP over Streamable HTTP — see "inady KANBAN MCP server" below). The ticket GET/PATCH/POST routes are what the **inady KANBAN MCP server** drives over either transport.
- The upgrade handler forwards **non-terminal** upgrades to Next's HMR via `app.getUpgradeHandler()` (which must be obtained **after** `app.prepare()`). Never `socket.destroy()` other paths or dev fast-refresh breaks.

### PTY lifecycle & connection ownership (`pty-registry.ts` + `handleTerminal` in `server.ts`)
- `startSession` spawns a new PTY or **attaches** to an already-running one (background). Returns `{ status, connId, replay }`; only the attached `connId` may `stdin`/`resize`/`kill`. A second tab is rejected while one attachment is live.
- **Detach on WS close** — navigating away does **not** kill the PTY; stdout while detached is buffered (scrollback cap in `agent-limits.ts`) and sent as `replay` on reconnect.
- `killSession` **deletes the registry slot synchronously**, then SIGTERM → SIGKILL after 2s. Also marks DB-only `running` rows ended (for HTTP kill without a live PTY).
- `proc.onExit` updates the DB and notifies the attached handler only if that proc still owns the slot.
- On boot: `sweepOrphanAgentProcesses()` (PID file from `agent-pid-store.ts`) → `sweepRunningSessions()` (stale DB `running` → `error`).
- Concurrent limit: default **20** live spawns (`INADY_KANBAN_MAX_CONCURRENT_AGENTS`); `createAgentSession` also checks DB `running` count inside its insert transaction.
- **React Strict Mode is disabled** (next.config.mjs) because the `<Terminal>` mount effect spawns a real PTY; the dev double-invoke would double-spawn/kill.

### Agent CLI invocation (`src/lib/agent-cli.ts` + `src/lib/cursor-agent.ts` + `src/lib/cline-agent.ts`)
- Every session is an **interactive TUI** driven by one of three CLIs, chosen per session (`agent_sessions.agent`: `"cursor"` | `"claude"` | `"cline"`). `AGENT_CLIS` holds each CLI's bin (env-overridable: `CURSOR_AGENT_BIN`, `CLAUDE_BIN`, `CLINE_BIN`), argv builder, trust-prompt regex/answer, and output filter. There is no headless/one-shot mode (removed deliberately). **cline is disabled by default** (`DEFAULT_AGENT_ENABLED` in agent-tools.ts) — it only appears once the user enables it in Settings → AI tools; cursor and claude stay on.
- The conversation UUID is **pre-issued** before the PTY launches (so `agent_sessions.agentSessionId` exists first):
  - **cursor**: `cursor-agent create-chat` in the ticket's `working_dir` (cursor-agent.ts); launch is `--resume <id> [--model <m>] "<prompt>"`, re-open is `--resume <id> [--model <m>]`.
  - **claude**: a locally generated UUID, pinned at launch via `--session-id <id> "<prompt>"`; re-open is `--resume <id>`.
  - **cline**: cline's `--id` is **resume-only** (there's no `create-chat` and no pinnable `--session-id`), so cline **mints its own id** on first launch — the initial launch carries **no** `--id`: `-i --auto-approve true -P cline-pass [-m <model>] [--thinking <effort>] "<prompt>"`. `-i` opens the interactive TUI (bare `cline "prompt"` is headless). `pty-registry` then **recovers** the new id from `cline history --json` (matching our exact wrapped prompt + cwd, newest — see `pickClineSession`/`captureClineSessionId` in cline-agent.ts) and **overwrites** the placeholder `agentSessionId` with it; re-open is `-i --auto-approve true -P cline-pass [-m] [--thinking] --id <captured-id>` (no prompt). Capture is best-effort/fire-and-forget: on timeout the placeholder stays and a later resume just starts a fresh conversation.
  In all three, the initial prompt is a **positional arg** the TUI auto-submits (no stdin pre-feed; survives a trust dialog) — and for cline, passing `--id` on that initial launch is exactly what would suppress the prompt (it'd try to resume nothing), which is why the initial launch omits it. **cursor** keeps per-command approval (no `--force` / bypass flags); **claude** launches with `--permission-mode auto`; **cline** launches with `--auto-approve true` (both per-run autonomy settings, re-applied on resume).
- **Model selection differs by CLI.** claude takes **separate** `--model`/`--effort`, pinned to the conversation (passed only at initial launch). cursor takes a **single combined** `--model <id>` (effort baked in — there's no cursor `--effort`); `create-chat` takes no model, so `--model` is **per-invocation** and is re-passed on **every** cursor launch incl. resume (else resume reverts to cursor's default). cursor validates the id at runtime. cline is a hybrid: a **combined clinepass model id** via `-m <id>` (e.g. `cline-pass/glm-5.2`) under a fixed `-P cline-pass` provider, **plus** a separate `--thinking <effort>` level (`none|low|medium|high|xhigh`); both are per-invocation and re-passed on every launch incl. resume (like cursor's `--model`). cline's effort enum + `DEFAULT_CLINE_EFFORT` live in agent-launch.ts.
- **Workspace trust**: cursor and claude prompt on untrusted dirs; `pty-registry` watches the stream for the CLI's `trustPromptRe` and auto-sends its `trustAnswer` once (`a` for cursor, Enter for claude); trust persists per-workspace after that. cursor's `filterStderr` drops known harmless noise lines without reflowing the stream; claude/cline output passes through verbatim. cline runs with `--auto-approve` so there's no trust prompt — its `trustPromptRe` is an inert never-match.
- **`--worktree`** (per-session opt-in, `agent_sessions.worktree`, initial launch only) is supported by all three CLIs. cursor/claude take an optional `[name]` so the flag leads the argv (else it swallows the prompt); cline's `--worktree` takes no arg and auto-creates a *detached* worktree under `~/.cline/worktrees/`. Resume never re-passes it (that would spawn a second worktree).
- The DB column behind `agentSessionId` is still physically named `cursor_session_id` — kept deliberately, since `db:push` has no migration files and a rename would be destructive.
- Session status: `running` / `finished` (exit 0) / `error` (non-zero or crash sweep) / `killed` (user-initiated stop — kill paths mark `killed`, not `error`).
- **Activity overlay** (`agent_sessions.activity`: `"busy" | "awaiting" | null`): orthogonal to `status` — a hook-reported view of a `running` session. It's **not** detected from output; the agent's own hooks report it. `startSession` injects `INADY_KANBAN_SESSION_ID` + `INADY_KANBAN_URL` into the PTY env so the user's `SessionStart`/`UserPromptSubmit` (→ `busy`) and `Stop`/`Notification` (→ `awaiting`) hooks can `POST …/activity/(busy|awaiting)`; `setSessionActivity` sets it (idempotent, publishes an activity `SessionEvent`) and `markEnded`/(re)start/sweep clear it to null. The UI maps a running session by activity: **busy** → spinner, **awaiting** → amber "your turn" badge, **null (no hook)** → the classic green dot (we don't guess) — `sessionVisual`/`sessionBadges` in agent-display.ts, `tallySessionCounts` in board-order.ts. So the spinner only appears once a hook confirms `busy`; without hooks the board keeps the original green/red status dots. (`activity` is nullable, so `db:push` adds it cleanly.) See README "Agent status: working vs. your turn".

### Data flow (board ↔ DB)
- Pages are Server Components reading SQLite directly with `export const dynamic = "force-dynamic"`; Server Actions (`src/actions/`) mutate and call `revalidatePath`.
- **Ticket create/read/update each have one source of truth** in `src/lib/ticket-core.ts` (relative imports, no `next/cache`, loads in both module graphs): `insertTicket` (create), `getTicket`/`listTickets` (read), `updateTicketFields` (update editable fields; tags stay separate via `setTicketTags`). The Server Actions wrap these with `revalidatePath` (`createTicket`→`insertTicket`, `updateTicket`→`updateTicketFields`+tags); the `POST /api/tickets` / `GET /api/tickets[/:id]` / `PATCH /api/tickets/:id` endpoints (server.ts) call them directly for external callers. Don't reintroduce a second insert/update path — the board UI, the HTTP API, and the MCP server must all go through these. `insertTicket`/`updateTicketFields` also `publishTicketEvent` on the shared bus (see "board event bus") so **other** open tabs auto-refresh — `revalidatePath` only updates the acting tab. Co-locating the broadcast in the one write path (rather than at each call site) is why it can't be forgotten.
- DnD: `Board.tsx` is a thin dispatcher — the ordering math is the pure `computeDragResult` in `src/lib/board-order.ts`, which returns either a `move` (fractional `position` = average of neighbors) or a `reorder` (integer renumber when the gap < ε). `Board` applies the result via `useOptimistic` + `moveTicket`/`reorderColumn`. Status labels live in `src/lib/ticket-display.ts`; agent labels/logos/status-dot colors in `src/lib/agent-display.ts` (client-safe — never put display data in agent-cli.ts, which pulls `node:child_process`).
- **Done column shows only the latest 10** (by `updatedAt` desc — `orderDoneColumn`); the page queries done tickets with `LIMIT 10` plus a total count. `computeDragResult` still requires position-sorted input, so `Board` keeps two groupings: `byStatus` (position order, for drag math) and `displayByStatus` (Done re-sorted by recency, render only). Within-Done drags are ignored; any drag *into* Done kills the ticket's agents first (both `move` and `reorder` shapes).
- **Ice Box is a count-only column** (status `icebox`): the board's active query excludes it (`notInArray(status, ["done","icebox"])`) and shows only `iceboxTotal` behind the droppable `IceBoxTile` (next to To Do). The drag math still groups over the *full* `TICKET_STATUSES`, so the empty `icebox` bucket exists and a drop onto the tile resolves through `computeDragResult` → `moveTicket(id,"icebox",pos)` — **no agent kill** (unlike Done; same parking behavior as WIP). `Board` renders `COLUMN_STATUSES` (`TICKET_STATUSES` minus icebox) as the draggable columns. The tickets themselves live on the **`/icebox`** page (Server Component, `force-dynamic`); each row's `IceBoxRowActions` (client island) revives via `moveTicketToTodo` (to the end of To Do) or deletes via `killTicketSessions` → `deleteTicket` (both revalidate `/` + `/icebox`). The board count is a plain server prop, so it updates a beat after a drop (`revalidatePath`), not optimistically.
- The **auto Doing transition** lives in `createAgentSession` (pre-issue conversation UUID → INSERT session `running` → set ticket `doing`, in one transaction) — "execution started" is the trigger, before the PTY runs. Resuming an *existing* session is the other "resume work" funnel: the first keystroke into a session terminal (`Terminal`'s `onFirstInput`) calls the `resumeTicket` action, which pulls a parked **WIP** ticket back into Doing (no-op for any other status). Triggering on first *input* — not on open — keeps WIP a glance-safe parking lot: merely opening/auto-restoring a session to read it doesn't un-park the ticket, but typing to the agent does. This covers both the explicit SessionList "open" click and the passive auto-open-on-load path, since both mount the same `Terminal`.
- SQLite foreign keys are enabled per-connection in `client.ts` (`PRAGMA foreign_keys = ON`), so deleting a ticket cascades its `agent_sessions`.
- **Repositories & editors are DB-backed, managed in Settings** (`repositories`, `editors` tables). DB readers (`readWorkingDirs`/`listRepositories`/`listEditors`) sit with the other list-readers in `src/lib/inady-kanban-config.ts`; CRUD is in `src/actions/{repositories,editors}.ts`. `readWorkingDirs` runs paths through the pure `parseWorkingDirs` (working-dirs.ts) — the same helper that normalizes the one-shot legacy-file import. The shared `assertValidWorkingDir` (absolute + existing dir) also lives in working-dirs.ts (deduped out of the ticket/template actions).
- **Boot-time defaults** (`bootstrapDefaults` in `src/lib/bootstrap.ts`, called from `server.ts` after `app.prepare()`): one-shot import of the legacy `data/working-dirs.json` into `repositories` + seed of the default `editors` (Cursor classic/VS Code/Emacs), guarded by `app_settings.seeded` plus per-table empty checks so clearing a list stays cleared. Errors are swallowed (logs a `db:push` hint) so the server still boots before the tables exist.
- **Cursor models are catalog + selection** (mirrors `agent-tools.ts`, no new table). The **catalog** — the full `{id,label}` set from `cursor-agent --list-models` — is the committed `src/lib/cursor-models.generated.json`, regenerated by `scripts/sync-cursor-models.ts` (`npm run sync:cursor-models`) and refreshed by a daily GitHub Action (`.github/workflows/sync-cursor-models.yml`) that opens a PR on change (needs a `CURSOR_API_KEY` repo secret). The **selection** — which ids to show, order, default — is JSON on `app_settings.cursor_models` (read/written via `readCursorModelSelection`/`writeCursorModelSelection` in app-settings.ts, saved by `saveCursorModels`); unset ⇒ `parseCursorModelSelection` returns Composer 2.5 + Auto (no seeding). Settings "Cursor models" curates it (reorder/default/remove + an "add" dropdown holding the long catalog). The pure helpers live in `cursor-models.ts`; do not give cursor a second model path.
- **Cline models** mirror the cursor selection layer (`src/lib/cline-models.ts`, same pure helpers + Settings "Cline models" UI + `app_settings.cline_models` via `readClineModelSelection`/`writeClineModelSelection`, saved by `saveClineModels`; unset ⇒ GLM-5.2). The one difference: there is **no `cline --list-models`** — the clinepass catalog is fetched from cline's API at runtime, so `CLINE_MODEL_CATALOG` is a **hand-maintained constant** transcribed from the clinepass docs (no generated json, no sync script/Action). Cline's effort is the orthogonal `--thinking` level (agent-launch.ts), not part of the model id; its **board-level default** lives on `app_settings.cline_effort` (default `xhigh`, editable in Settings → "Cline default effort" via `readClineDefaults`/`writeClineDefaults`, saved by `saveClineDefaults`) — mirrors the claude `claude_effort` default. New sessions seed from it (`createAgentSession` falls back to it when no effort is passed) and can still override per-launch; the model default stays in the `cline_models` selection.
- **macOS folder picker**: `pickRepositoryDirectory` (server action) shells out to `osascript`'s `choose folder`; the pure output cleaner is `cleanChosenFolderPath` (working-dirs.ts, unit-tested). Non-darwin throws a friendly message and the UI falls back to direct path entry.
- **"Open with"** (`openInEditor` server action, `OpenWithButton` split button on the ticket page): runs the editor's command via `spawn(shell:true, detached, stdio:"ignore")` with `cwd` = the ticket's working dir (so `.` resolves there; the path is never interpolated — fine for this localhost single-user tool). Editors keep an **exactly-one-default** invariant enforced in transactions (saveEditor/setDefaultEditor/deleteEditor); `pickDefaultEditor` (editor-commands.ts, client-safe + tested) returns the flagged default else the first by order.

### inady KANBAN MCP server (`src/lib/inady-kanban-mcp.ts` + two transports)
A local, no-auth MCP server (`@modelcontextprotocol/sdk`) that lets a coding agent create/update/read tickets, exposed over **two transports that share one tool set**: the board's own HTTP endpoint and a stdio subprocess. The four tools are registered once in the `createMcpServer()` factory (`src/lib/inady-kanban-mcp.ts`); both transports build from it, so they cannot drift apart.
- **HTTP (preferred)**: `server.ts` mounts `/mcp` over `StreamableHTTPServerTransport` in **stateless** mode (a fresh server+transport per request — `sessionIdGenerator: undefined`, `enableJsonResponse: true`; the transport reads the body itself, so don't pre-consume it; non-POST → 405). So while the board runs (`npm run dev`), MCP is live at `http://localhost:7373/mcp` with no extra process; the in-process tools call back through loopback (`serverBaseUrl()`).
- **stdio**: `mcp-server.ts` (`npm run mcp`, `tsx mcp-server.ts`) wires the same factory to `StdioServerTransport`; needs the board running; target URL is `$INADY_KANBAN_URL` (else the local server). Stdio means **nothing may go to stdout except protocol frames** — diagnostics use `console.error`.

**It holds no ticket logic** — each tool is a thin call through `src/lib/inady-kanban-mcp-client.ts` (a `fetch`-based HTTP client) to the board's own `/api/tickets` endpoints, which call ticket-core. So MCP (both transports), the HTTP API, and the board UI share **one** implementation — this is the load-bearing design rule for this feature; don't give the MCP its own DB/logic path. The client is the only unit-tested piece (pure, `fetch` injected — same philosophy as the other `src/lib` helpers); the factory, `mcp-server.ts`, and the server.ts `/mcp` route are integration, verified by running. Tools: `inady_kanban_list_tickets`, `inady_kanban_get_ticket`, `inady_kanban_create_ticket`, `inady_kanban_update_ticket`. See README "Let agents manage the board (MCP)".
